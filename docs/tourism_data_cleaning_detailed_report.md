## 4. 数据清理原则

本次清理遵循以下原则：

1. 不修改原始 Excel 文件。
2. 不随意填补主变量缺失值，避免人为制造趋势。
3. 月度分析优先保留月度频率变量，年度变量只作为补充信息。
4. 聚类分析和决策树分析使用单独准备过的衍生字段，避免不同模型互相干扰。
5. 所有清理动作都应当能够解释“为什么这样做”，而不是只追求技术上的完整。

## 5. 具体清理内容

### 5.1 删除元数据行

**做了什么：**

- 删除了原始工作表前 29 行的元数据。

**为什么这么做：**

原始表前 29 行包含字段说明、频率、单位、数据来源、统计摘要等内容，这些内容不是逐月观测值，不能直接进入分析模型。如果不删除，这些文本行会被误识别为数据，导致类型转换失败或模型结果错误。

**影响：**

- 保留的才是实际时间序列观测值。
- 数据结构从“说明 + 数据混合表”变成“纯观测值表”。

### 5.2 统一字段结构与变量命名

**做了什么：**

将原始表中的 8 个核心变量整理为统一字段名：

- `date`
- `avg_stay_monthly`
- `hotel_occ`
- `spend_per_capita`
- `tourism_receipts`
- `avg_stay_annual`
- `visitor_arrivals`
- `visitor_arrivals_china`

**为什么这么做：**

原始 Excel 的列名较长，且夹杂说明性文本，不利于在 RStudio 中反复调用。统一字段名后更适合写代码、画图和建模，也能减少因为字段拼写不一致导致的错误。

**影响：**

- 提高后续分析的一致性和可重复性。
- 更方便写 R 代码和论文方法部分。

### 5.3 转换数据类型

**做了什么：**

- 将 `date` 转换为日期型变量。
- 将其余分析变量转换为数值型变量。

**为什么这么做：**

Excel 读入后，日期和数值可能被识别成字符或混合类型。如果不先转换，时间序列图、分组统计、相关分析和决策树建模都会出现错误。

**影响：**

- 变量可以被正确排序、计算、建模和可视化。

### 5.4 删除关键主变量缺失的无效月份

**做了什么：**

- 删除了 `2015-12-01` 这一条记录。

**为什么这么做：**

这一行在月度主分析变量 `avg_stay_monthly`、`hotel_occ`、`visitor_arrivals`、`visitor_arrivals_china` 上都是缺失值。虽然该行保留了年度变量，但它无法参与月度趋势分析、聚类或决策树，因此继续保留只会在后续处理中制造缺失噪音。

**影响：**

- 原始数据区从 `111` 行变为 `110` 行。
- 被删除的日期只有 `2015-12-01` 一行。

### 5.5 保留年度变量，但不强行插补

**做了什么：**

- 保留了 `spend_per_capita`、`tourism_receipts`、`avg_stay_annual`。
- 没有对这些年度变量做月度插值或前向填充。

**为什么这么做：**

这些变量本身就是年度频率，只有年末月份存在观测值。如果把它们强行扩展到每个月，会把“年度指标”误写成“月度指标”，这会导致 EDA 结论被夸大，也会使聚类和决策树的时间频率失真。

**影响：**

- 这些变量仍可用于年度补充图表和描述性讨论。
- 它们不会污染月度模型。

清理后年度变量缺失情况如下：

- `spend_per_capita` 缺失 `101` 个值
- `tourism_receipts` 缺失 `101` 个值
- `avg_stay_annual` 缺失 `100` 个值

这不是清理失败，而是由变量频率决定的正常结果。

### 5.6 新增日历变量

**做了什么：**

新增了以下时间辅助变量：

- `year`
- `month`
- `quarter`

**为什么这么做：**

这类变量是后续分组分析和模型解释的基础：

- `month` 可用于观察季节性
- `quarter` 可用于较粗粒度的分组
- `year` 可用于年际变化分析

**影响：**

- 便于按月份做箱线图、按季度做分组统计、按年份查看恢复节奏。

### 5.7 新增疫情阶段变量 `period`

**做了什么：**

新增了 `period`，将每个月份划分为 3 个阶段：

- `pre_covid`：截至 `2020-01`
- `covid_shock`：`2020-02` 至 `2021-12`
- `recovery`：`2022-01` 起

阶段样本数如下：

- `pre_covid`：`38`
- `covid_shock`：`23`
- `recovery`：`49`

**为什么这么做：**

本研究的核心就是分析疫情冲击与复苏，因此需要一个可以直接分组比较的阶段变量。这样做后，EDA 和 CDA 可以很自然地围绕三个阶段比较差异。

**影响：**

- 可直接用于箱线图、显著性检验和分组统计。
- 有利于解释旅游市场在不同阶段的结构差异。

### 5.8 新增中国游客占比变量

**做了什么：**

新增了：

- `china_share = visitor_arrivals_china / visitor_arrivals`
- `china_share_pct = china_share * 100`

**为什么这么做：**

如果只使用中国游客绝对人数，模型更多反映的是“规模变化”；而加入中国游客占比后，就能分析“中国市场在整体旅游中的权重变化”。这对聚类和决策树都更有解释力。

**影响：**

- 可以直接比较不同月份中国市场的重要性。
- 可用于解释“市场结构变化”而不仅仅是“总量变化”。

### 5.9 对平均逗留时间进行高端截尾

**做了什么：**

对 `avg_stay_monthly` 的极高值进行了高端截尾，生成新变量：

- `avg_stay_monthly_capped`

截尾上限设为 `95%` 分位数，即：

- 截尾值：`33.831658`

共有 `6` 条记录被截尾。

**为什么这么做：**

疫情冲击期出现了极端高逗留时间值。这类值从业务上并非错误，但它们会在聚类分析和决策树中占据过强权重，使模型过度围绕少数极端月份分裂，而不是识别更一般性的市场规律。

因此，这里采用“保留原值 + 另建截尾变量”的方式：

- `avg_stay_monthly` 保留原始信息，适合描述性分析
- `avg_stay_monthly_capped` 适合建模

**影响：**

- 降低极端值对模型稳定性的干扰
- 保留原始变量，避免信息丢失

### 5.10 为聚类分析生成标准化字段

**做了什么：**

新增了 4 个聚类专用标准化字段：

- `cluster_z_visitor_arrivals`
- `cluster_z_china_share`
- `cluster_z_hotel_occ`
- `cluster_z_avg_stay_monthly_capped`

这些字段采用 z-score 标准化处理。

**为什么这么做：**

聚类分析基于距离计算，不同变量量纲差异很大：

- 旅客人数是几十万到上百万
- 入住率是百分比
- 逗留时间是天数
- 占比是 0 到 1 之间

如果不先标准化，距离会被“人数”这种大数值变量主导，导致聚类结果偏向单一维度，不利于识别真正的市场状态差异。

**影响：**

- 让多个变量在聚类时权重更平衡
- 提高聚类结果的可解释性

### 5.11 为决策树生成分类标签

**做了什么：**

新增了两个目标变量：

- `hotel_occ_level_tertile`
- `hotel_occ_level_business`

其中：

1. `hotel_occ_level_tertile`  
   按入住率三分位数划分为 `low`、`medium`、`high`

   三分位切点为：

   - 低位切点：`77.983170`
   - 高位切点：`84.150924`

   类别分布为：

   - `low`：`37`
   - `medium`：`36`
   - `high`：`37`

2. `hotel_occ_level_business`  
   按业务阈值划分：

   - `< 70` 为 `low`
   - `70 - 85` 为 `medium`
   - `> 85` 为 `high`

   类别分布为：

   - `low`：`25`
   - `medium`：`56`
   - `high`：`29`

**为什么这么做：**

决策树分析如果直接预测连续型入住率，可以做回归树；但在课程作业中，分类树通常更好解释。通过把入住率离散化成高、中、低三类，结果可以直接写成“哪些条件下月份更可能落入高入住率状态”。

设置两个版本的原因是：

- `tertile` 版本类别更均衡，更适合建模
- `business` 版本更贴近业务理解，更适合展示解释

**影响：**

- 提高决策树结果的可读性
- 允许你根据老师偏好选择更统计化或更业务化的版本

### 5.12 按时间顺序划分训练集和测试集

**做了什么：**

新增了 `dataset_split` 字段：

- `train`：`87` 条
- `test`：`23` 条

划分方式为按时间顺序的 `80/20` 切分，而不是随机切分。

**为什么这么做：**

本研究数据是时间序列月度数据。如果随机划分训练集和测试集，未来月份的信息可能会“泄漏”到训练中，从而高估模型效果。按时间顺序切分更符合真实预测场景，也更符合时间序列分析逻辑。

**影响：**

- 提高模型验证的合理性
- 避免随机抽样带来的时间泄漏

## 8. 总结

本次数据清理并没有对原始数据做激进修改，而是围绕“保证 4 个分析模块可以顺利开展”进行了有针对性的整理。核心思路可以概括为：

1. 删掉不是观测值的部分  
2. 保留真正可用于月度分析的主变量  
3. 不强行填补不同频率变量  
4. 为聚类和决策树单独准备衍生字段  
5. 在保留原始信息的同时，提高模型稳定性和解释性

因此，最终得到的数据既可以支持可视化和统计检验，也可以直接用于聚类分析和决策树分析。
